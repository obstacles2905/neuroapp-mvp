# Замер осанки / позы (фото тела + MediaPipe)

Документ фиксирует **бизнес-смысл** и **техническую архитектуру** замера «геометрия тела» для дальнейшей разработки. Обзор сторонних библиотек см. файл со стеком в `docs/` (например `ba-biometry.md`).

---

## 1. Бизнес-логика (кратко)

- Замер — **опора протокола**: базовая точка, динамика во времени, возможная персонализация уроков; не декоративная фича.
- Пользователь проходит **протокол съёмки** (ракурс, дистанция, подсказки), чтобы повысить воспроизводимость.
- На выходе нужны не «сырые точки нейросети», а **осмысленные метрики методики** (углы, асимметрии, агрегаты) + **оценка качества** кадра/сессии.
- **Приватность по умолчанию:** тяжёлый анализ **на устройстве**; пиксели не отправляются на сервер; на бэкенд при необохдимости — только **агрегаты** (метрики, версия модели, quality flags, идентификатор замера).

---

## 2. Три слоя данных (контракт мышления)

| Слой | Содержимое |
|------|------------|
| **Наблюдение** | Кадр(ы), метаданные съёмки, пермишены, таймстемп. |
| **Инференс** | Лендмарки MediaPipe, visibility/presence, **версия модели** (`modelId` / `modelVersion`). |
| **Продукт** | Бизнес-метрики, статусы качества (`ok` / retry / failed), тексты для UI. |

Смена модели или формул не должна ломать сравнимость без учёта: в каждой записи замера хранить **версию модели** (и при необходимости сырые лендмарки для пересчёта метрик).

---

## 3. Захват: серия кадров (burst), не одно фото

- Один кадр сильнее шумит от случайной позы, света, микродвижения.
- **Целевой UX:** один проход пользователя («встал в рамку»), **короткая серия снимков** с малой задержкой.
- **Агрегация:** напр. выбор «лучшего» кадра по полноте видимости точек, медиана/робастные оценки углов по кадрам с `confidence > порог`.
- Архитектурно заложить абстракцию **`FrameSource`**: `single_image` | `burst` | (в перспективе) короткий клип — чтобы не переписывать продуктовый поток при смене источника.

---

## 4. Техническая архитектура в репозитории (Expo / React Native)

### 4.1. Где живёт MediaPipe

- MediaPipe Pose Landmarker — **нативный** конвейер (C++ / Tasks API, модель `.task` и т.д.).
- **JavaScript не считает позу** — только оркестрация UI, вызов моста, хранение результата.

### 4.2. Expo

- Кастомная нативка **не работает в Expo Go**; целевой цикл: **development build** (например EAS) + `prebuild` при необходимости.
- Текущее приложение: каталог `mobile/` (Expo SDK 54, включён new architecture — учитывать при создании модуля).

### 4.3. Модульность в монорепо

| Вариант | Назначение |
|---------|------------|
| **Локальный Expo Module** (рекомендуется) | Инкапсуляция MediaPipe: загрузка модели, inference, стабильный JSON-контракт в JS. Пример путей: `mobile/modules/neuro-pose-landmarker/` или `packages/neuro-pose-landmarker/`. |
| **Микросервис с MediaPipe на бэкенде** | Не базовый путь: конфликт с приватностью, задержка, стоимость; возможен только как отдельный режим с явным согласием. |
| **Правки только в `android/` / `ios/` без модуля** | Быстро ломается при повторных `prebuild`; не рекомендуется как долгосрочная стратегия. |

### 4.4. JS-слои приложения

- **`features/biometrics/`** — сценарии (онбординг, таб «Био», урок): UI, пермишены, burst-захват, вызов нативного модуля, отображение результата, локальное сохранение записи замера.
- **Нативный модуль** — без копирайтов и бизнес-правил уроков; только inference и маппинг в контракт лендмарков.
- **Абстракция `CaptureAdapter`** — позволяет поменять реализацию камеры (например `expo-camera` → Vision Camera + frame pipeline) без переписывания бизнес-потока.

### 4.5. Граница с `backend/`

- Бэкенд хранит домен времени: **метрики**, идентификатор замера, версия модели, quality, опционально очередь синка.
- **Сырые фото** на сервер по умолчанию **не загружаются**.

---

## 5. Нефункциональные требования (тезисно)

- **Cold start:** ленивая загрузка модели при первом входе в био-раздел; индикатор «подготовка».
- **Ошибки:** провалы ракурса/света → понятные инструкции «как исправить», а не общий «сбой анализа».
- **Доступность:** деградация, если полный силуэт недоступен (альтернативный сценарий или честный отказ).
- **Валидация качества продукта:** золотой набор кадров для регрессии при смене модели; контроль дрейфа метрик.

---

## 6. Этапы внедрения (ориентир)

1. Протокол съёмки + burst-захват + базовые эвристики качества кадра (до или вместе с ML).
2. Нативный модуль + MediaPipe + версионирование + сохранение лендмарков локально.
3. Первые бизнес-метрики и экран результата.
4. График / привязка к урокам; синк агрегатов на бэкенд при готовности API.
5. Оптимизация источника кадра (FPS, лучший кадр из короткого видео) при сохранении того же UX.

---

## 7. Связанные файлы в репозитории (по мере появления кода)

- Продуктовые экраны: `mobile/app/...`, `mobile/features/biometrics/`.
- Захват серии: `mobile/features/biometrics/PostureBurstCapture.tsx`.
- Пайплайн и метрики: `mobile/lib/biometrics/run-pose-pipeline.ts`, `compute-pose-metrics.ts`, `mediapipe/detect-pose-on-image.ts` (вызов TurboModule **без** импорта entry пакета, чтобы не тянуть VisionCamera frame processor).
- Модель MediaPipe (`pose_landmarker_lite.task`): `mobile/assets/models/`; в `app.json` подключён config plugin `react-native-mediapipe-posedetection`.
- **Сборка:** нативные зависимости не входят в Expo Go — нужны `npx expo prebuild` и dev client / EAS Build.
- Справочник стека: [`docs/ba-biometry.md`](./ba-biometry.md) (или актуальное имя файла со стеком).

*При существенных решениях (облачный inference, хранение сырых фото) — обновлять этот документ и явно фиксировать отклонение от базовой модели «всё на устройстве».*
