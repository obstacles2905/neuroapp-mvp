# Архитектурно-бизнесовая спецификация: Переход на MND-Protocol

## 1. Контекст для AI-ассистента (Cursor)
На данный момент наша система работает по статичной модели: `Категория -> Статичный список уроков -> Упражнения`. Пользователь выбирает категорию и проходит заготовленные уроки.

**Новая бизнес-задача:** Переход на динамическую систему "MND-Protocol" (Matrix of Neurophysiological Deficits). Уроки больше не являются жестко привязанными к категориям. Программа (сессия) для пользователя должна **генерироваться динамически** (как плейлист) на основе его индивидуальных симптомов, математических весов и мастер-стеков упражнений.

Твоя задача — спроектировать технические решения и плавно мигрировать текущую кодовую базу на эту логику, сохраняя уже написанные механизмы (drag-and-drop, zustand, server actions и т.д.), но изменив связи между сущностями. Выбор конкретных таблиц, БД и эндпоинтов остается за тобой.

---

## 2. Ключевые бизнес-сущности (Domain Model)

Тебе необходимо абстрагироваться от старой модели и реализовать следующие концептуальные сущности:

1. **Симптом (Symptom):** То, на что жалуется пользователь (например, "Жвачка в голове", "Сжатая пружина"). Заменяет собой старую сущность "Категория/Проблематика" на этапе онбординга.
2. **Мастер-стек (Master Stack):** Глобальные "корзины" (ST-1, ST-2 ... ST-6), в которых лежат все упражнения. Упражнения больше не принадлежат одному уроку или симптому. Они принадлежат Стеку.
3. **Упражнение (Exercise / Step):** Единица контента. Каждое упражнение в стеке обязательно имеет тег направленности: либо **Bottom-Up** (от тела к мозгу), либо **Top-Down** (от мозга к телу).
4. **MND-Матрица (Lookup Table):** Связующее звено (настройка). Связывает конкретный Симптом с:
   - *Пропорцией направленности:* % Bottom-Up против % Top-Down (например, 80/20).
   - *Приоритетными стеками:* из каких Мастер-стеков брать упражнения для этого симптома (например, ST-2 и ST-1).
5. **Персональный Протокол (Генератор Сессий):** Движок, который собирает "плейлист" упражнений на сегодня/неделю, учитывая все выбранные пользователем Симптомы, усредняя их веса и перемешивая упражнения.

## 3. Поэтапный план разработки (Iterative Roadmap)

### Этап 1: Адаптация Админ-панели (Управление MND и Стеками)
*Цель: Дать контент-менеджеру возможность заносить данные по новой логике.*
- **Сущности:** Создать управление Мастер-стеками (ST-1...ST-6) и Симптомами (20 базовых симптомов).
- **Связи:** Реализовать интерфейс настройки MND-Матрицы (привязка Симптома к % соотношения B-Up/T-Down и приоритетным Стекам).
- **Упражнения (Lesson Builder):** Добавить возможность при создании упражнения (шага) указывать, к какому Мастер-стеку оно относится и какой у него тип (Bottom-Up / Top-Down).
- **Миграция:** Отвязать существующие сущности уроков от жестких категорий.

### Этап 2: Трансформация Онбординга (Backend + Mobile)
*Цель: Пользователь должен выбирать Симптомы, а не Категории.*
- **Бэкенд:** Реализовать отдачу списка Симптомов для онбординга.
- **Мобилка:** Переделать экраны онбординга — вместо категорий выводить карточки Симптомов (можно с описанием "Суть", чтобы юзер узнавал себя).
- **Сохранение:** Сохранять выбранные Симптомы в профиль пользователя (с возможностью в будущем менять их актуальность).

### Этап 3: Движок "Алгоритм Взвешенного Резонанса" (Backend)
*Цель: Написать алгоритм динамической генерации тренировки (Session Generator).*
- **Логика алгоритма:** 1. Получить все актуальные Симптомы пользователя.
  2. Найти их настройки в MND-Матрице.
  3. Если Симптомов несколько — найти пересечения по приоритетным Мастер-стекам и высчитать средний % для Bottom-Up и Top-Down.
  4. Сгенерировать сессию (например, из 6 упражнений), где N упражнений — это Bottom-Up из приоритетных стеков, а M упражнений — Top-Down.
  5. **Правило ротации:** Добавить механизм shuffle, чтобы алгоритм чередовал упражнения внутри стека и не выдавал одно и то же каждый день.
- *Техническое требование:* Оформить это как отдельный сервис генерации, который можно будет переиспользовать.

### Этап 4: Адаптация Мобильного приложения (Выполнение протокола)
*Цель: Обучение становится персонализированным потоком.*
- **API:** Вместо запроса статического урока из категории, мобилка запрашивает "Сгенерировать сессию на сегодня".
- **UI Таба "Учеба":** Отображать карточку "Ваша персональная сессия" вместо каталога.
- **Плеер уроков:** Текущий механизм прохождения уроков (шаг за шагом) должен уметь принимать на вход динамически собранный массив шагов (упражнений) от бэкенда.
- **Прогресс:** Сохранять прогресс выполнения сгенерированной сессии (обновление стрика "дней подряд").

### Этап 5: "Тревожная кнопка" и "Джем-сессия" (Инкапсуляция алгоритма)
*Цель: Быстрая помощь в моменте.*
- **Тревожная кнопка (SOS):** Пользователь выбирает 1 острый симптом прямо сейчас (например, "Взрыв на ровном месте"). Бэкенд пропускает его через тот же генератор (Этап 3), но только для одного симптома, и выдает короткий плейлист (2-3 упражнения) для немедленного выполнения.
- **UI:** Вывести эту функцию на главный экран мобильного приложения.
